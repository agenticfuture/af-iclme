# Create Service Implementation Plan v1.1.0

## Scope
Stabilize and verify end-to-end Create Service flow with real persistence, job execution, deployment status, and tenant-facing bot runtime validation through playground integration.

## Phase 1 (Product)
1. Keep step-wise wizard persistence (`service`, `chatbot`, `data-sources`) with idempotent saves.
2. Ensure finish flow triggers deploy and redirects to playground route.
3. Validate DB persistence for:
   - `service.services`
   - `job.jobs`
   - `service.deployments`
4. Validate bot stream bridge returns real SSE chunks through `/api/chat`.

## Phase 2 (Security)
1. Enforce realm/tenant/org scoping in create-service and data-source paths.
2. Keep role checks for write operations (save/update/delete/process/deploy).
3. Ensure bot runtime cookie security works behind HTTPS reverse proxy.
4. Keep iframe isolation and restrictive headers on SaaS shell.

## Phase 3 (Privacy)
1. Enforce file-source object path scope checks before job enqueue.
2. Enforce existence checks for object paths in MinIO.
3. Preserve atomic delete behavior (`MinIO delete` + DB row delete).
4. Keep PII redaction checks in ingestion/privacy suites.

## Phase 4 (Operational Runtime)
1. Ensure `bot.servicegen.local` public origin is correctly configured.
2. Ensure guest/auth redirects do not create loops under HTTPS+Traefik.
3. Keep chat/embed surfaces public for playground/embed use cases.
4. Verify bot runtime health endpoint and streaming endpoint continuously.

## Phase 5 (Scoped Embed + Anonymous Isolation)
1. Add signed embed scope token (`sg_scope_token`) generated by SaaS for:
   - `tenant_id`
   - `org_id`
   - `service_id`
   - `access_scope` (`playground` | `public_embed`)
   - `exp` / `iat`
2. Enforce strict scope validation in bot runtime:
   - validate token signature + expiry,
   - reject missing/invalid scope ids,
   - reject any body/query scope mismatch against token claims.
3. Refactor chatbot DB persistence to store scope in chat/message records:
   - `Chat`: `tenant_id`, `org_id`, `service_id`
   - `Message_v2`: `tenant_id`, `org_id`, `service_id`
4. Persist both user input and assistant streamed output to chatbot DB with scope.
5. Anonymous user model:
   - per-session anonymous actor identity (not shared global anonymous user),
   - anonymous actor cookie binding,
   - scoped retrieval/write isolation using `tenant/org/service`.

## Verification Strategy
1. API suites:
   - `test_create_service_flow_contract.py`
   - `test_create_service_security_controls.py`
   - `test_create_service_privacy_controls.py`
2. SaaS e2e:
   - `create-service-playground-flow.spec.ts`
3. Bot live stream:
   - `POST https://bot.servicegen.local/api/chat` with real tenant/org/service context.
4. DB evidence:
   - Query latest service/job/deployment rows and statuses.
5. Scoped embed evidence:
   - token tamper/expiry rejection,
   - mismatch rejection (`query/body` vs token claims),
   - chat/message rows persisted with exact `tenant/org/service`,
   - anonymous sessions are isolated per session identity.

## Exit Criteria
1. All create_service API suites pass.
2. SaaS wizard e2e passes in rerun-stable state.
3. Bot stream endpoint returns valid SSE sequence with real text deltas.
4. Deployment and service rows are verifiable in DB with expected status transitions.
5. Embed bot accepts only signed scoped requests and rejects tampered scope input.
6. Chatbot DB stores user+assistant messages with tenant/org/service scope fields.
